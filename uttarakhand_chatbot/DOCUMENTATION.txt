--------------------------------------------------------------------------------
1. PROJECT OVERVIEW
--------------------------------------------------------------------------------
The Uttarakhand Heritage Chatbot is an independent NLP-driven service integrated
into the Uttarakhand GI Tag products website. It allows users to ask questions
about Uttarakhand's geographical indications (GI tags), local culture, cuisine,
festivals, wildlife, geography, and history. 

Features:
- Responsive standalone web chat page (/chat).
- Speech-to-text voice search capabilities via Web Speech API.
- Semantic matching against a local knowledge base.
- Dynamic fallback live web scraping for query expansion.
- Local JSON file-based cache manager for near-instantaneous query re-runs.
- Complete offline-first NLTK intent matching, TF-IDF vectorizer, and 
  Cosine Similarity scoring (no paid LLM APIs used).

3. TECHNOLOGIES USED
--------------------------------------------------------------------------------
- FastAPI: Highly performant, modern, asynchronous web framework for building APIs in Python.
- NLTK (Natural Language Toolkit): Standard library for human language processing, used for tokenization, stopword removal, and word stemming.
- scikit-learn (TF-IDF Vectorizer): Translates plain-text documents and queries into mathematical vectors based on word frequencies.
- Cosine Similarity: Used to calculate the angular distance between vectors to identify semantic overlap.
- BeautifulSoup4: Python library for pulling data out of HTML, used as a backup parser.
- Trafilatura: Main high-performance web extraction tool that cleans boilerplates, sidebars, and ads from pages.
- Requests: HTTP client library for downloading raw web pages.
- Cache System: Local filesystem-based caching of queries to prevent repetitive web queries and improve response speed.

--------------------------------------------------------------------------------
4. CHATBOT WORKFLOW
--------------------------------------------------------------------------------
When a user sends a query:

  User Query
      ↓
  Cache Check
      ├─→ [Hit]  → Retrieve Cached JSON Response → Return
      └─→ [Miss] → Continue
      ↓
  Intent Detection
      ├─→ [Not Relevant] → Return Rejection Message ("Not a valid Uttarakhand tag.")
      └─→ [Relevant]     → Continue
      ↓
  Semantic Search
      ├─→ [Match Found >= Threshold (0.3)] → Generate Response → Cache → Return
      └─→ [Match Found < Threshold]        → Continue
      ↓
  Web Scraping (Fallback)
      ├─→ DuckDuckGo search for: "{subject} {tag} Uttarakhand"
      ├─→ Fetch web pages → Extract and clean main content
      ├─→ Save to Local Knowledge Base & Re-index Search Engine
      ├─→ [Scrape Success] → Generate Response → Cache → Return
      └─→ [Scrape Failure] → Return fallback "not found" response

--------------------------------------------------------------------------------
5. INTENT DETECTION
--------------------------------------------------------------------------------
The intent detection module (intent_detector.py) processes the query through three main NLP steps using NLTK:
1. Tokenization: Splits queries into individual word units using NLTK's word_tokenize.
2. Stopword Removal: Filters out filler words (e.g., "is", "a", "the", "about", "tell", "me") that do not carry semantic intent.
3. Stemming: Reduces words to their base form (stems) using the Porter Stemmer (e.g., "festivals" -> "festiv", "cooking" -> "cook").

Once cleaned, the stems are matched against a tag keyword dictionary defined in config.py. If enough overlapping terms or general Uttarakhand keywords are found, the intent is classified as relevant, and the most matched category tag is extracted. Unrelated queries fail the relevance check and are immediately rejected.

--------------------------------------------------------------------------------
6. SEMANTIC SEARCH
--------------------------------------------------------------------------------
Local search operates in search_engine.py using TF-IDF (Term Frequency-Inverse Document Frequency) and Cosine Similarity:
- TF-IDF Vectorizer: Converts every article description in the knowledge base, and the incoming query, into a sparse numerical vector.
  - Term Frequency: How often a word occurs in a document.
  - Inverse Document Frequency: How unique a word is across all documents. Words like "Uttarakhand" have high document frequency, lowering their weight, whereas "Kafuli" or "Harela" have low document frequency, raising their weight.
- Cosine Similarity: Computes the cosine of the angle between the query vector and each document vector. A score of 1.0 represents perfect directional alignment, while 0.0 represents no overlapping terms.
- Threshold: If the highest similarity score meets or exceeds config.SIMILARITY_THRESHOLD (default 0.3), it is selected.

Example:
- Query: "recipe for kafuli"
- Doc A: "Kafuli is a traditional food made from spinach..." (Cosine similarity ~0.55)
- Doc B: "Nainital is a famous lake district..." (Cosine similarity ~0.05)
- Match: Doc A (exceeds 0.3) -> Selected!

--------------------------------------------------------------------------------
7. WEB SCRAPING
--------------------------------------------------------------------------------
When a query is relevant but no local article matches it, scraper.py triggers:
1. Web Search: Submits a POST request to DuckDuckGo HTML search for "{Subject} {Tag} Uttarakhand" and extracts the top organic links.
2. Fetching: Downloads raw HTML from the target URL using standard requests with a randomized User-Agent header and a timeout.
3. Content Extraction: Runs trafilatura to extract clean body text, automatically deleting navigation menus, headers, sidebars, and ads. If trafilatura fails, BeautifulSoup decomposes noise elements (<script>, <style>, etc.) and extracts text from paragraph tags.
4. Data Cleaning: Normalizes Unicode characters to NFC form, strips remaining HTML tags, collapses redundant whitespace, and truncates the content to 1000 characters.
5. Persistence: Saves the new entry (name, tag, description, source="web_scrape", source_url, timestamp) into data/knowledge_base.json and signals the Search Engine to reload and re-index.

--------------------------------------------------------------------------------
8. CACHE SYSTEM
--------------------------------------------------------------------------------
To maintain high performance and prevent repetitive web scraping:
- Hashing: The query string is lowercase-normalized, stripped of whitespace, and hashed using MD5.
- Filesystem Storage: Cached responses are stored as individual JSON files named after their MD5 hash in the `cache/` directory.
- Expiration (TTL): Each cache file saves a creation timestamp. If a cache request occurs after `config.CACHE_TTL_HOURS` (default 24 hours), the cache manager flags it as expired, deletes the file, and performs a fresh lookup.
- Shutdown: On FastAPI shutdown, the server runs a cleanup job that purges all expired cache files.

9. API DOCUMENTATION
--------------------------------------------------------------------------------
The chatbot runs on http://localhost:8001.

POST /chat
  Description: Process a query and return the matching response.
  Content-Type: application/json
  
  Request Format:
    {
      "query": "string (1-500 characters)"
    }
    
  Response Format:
    {
      "response": "string",
      "tag": "string | null",
      "source": "string ('cache', 'knowledge_base', or 'web_scrape')",
      "source_url": "string | null"
    }

  Example Request:
    POST http://localhost:8001/chat
    {
      "query": "Tell me about Kafuli"
    }
    
  Example Response:
    {
      "response": "Kafuli is a traditional dish from the Garhwal region of Uttarakhand...",
      "tag": "food",
      "source": "knowledge_base",
      "source_url": null
    }

10. FRONTEND ARCHITECTURE
--------------------------------------------------------------------------------
- Component: standalone Chat page is located at app/chat/page.tsx. It renders a dual-pane layout (left sidebar displaying active tags, main chat dashboard).
- Hooks: 
  - useEffect: Initializes webkitSpeechRecognition API on mount, manages auto-scrolling on chat updates.
  - useState: Manages chat message log, inputs, loading spinner, and mic listening indicators.
- API Client: lib/chatbot-api.ts handles HTTP posts to localhost:8001/chat. It wraps fetches in client-side AbortControllers to enforce a 15-second timeout, and uses an exponential backoff retry loop (maximum 3 retries) on server or network failures.
- State Management: The chat history is maintained in React component state (`messages` array), meaning it persists for the duration of the browser tab session.

--------------------------------------------------------------------------------
11. HOW TO RUN
--------------------------------------------------------------------------------
Backend Service:
1. Open a terminal and navigate to the chatbot directory:
   cd uttarakhand_chatbot
2. Install Python dependencies:
   python -m pip install -r requirements.txt
3. Start the FastAPI server:
   python main.py
   (The server will bind to http://localhost:8001)

Frontend Application:
1. Open a separate terminal in the project root:
   cd .. (if inside uttarakhand_chatbot)
2. Install Node dependencies:
   npm install
3. Launch the development server:
   npm run dev
4. Open your browser and navigate to:
   http://localhost:3000/chat
